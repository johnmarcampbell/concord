# 0019 — Mirror tables vs. record tables (amends 0002)

**Status**: Accepted, 2026-05-30. Amends [ADR 0002](0002-jsonl-as-canonical-raw-store.md).

## Context

[ADR 0002](0002-jsonl-as-canonical-raw-store.md) made JSONL the canonical raw store and declared every SQLite table "derived and rebuildable from this file." That framing was exactly right for the tables that existed then — `proceedings`, `chunks`, the FTS5 / vec indexes — and it remains the right default. Re-deriving SQLite from JSONL must stay a fast, free, lossless operation.

But the framing has been quietly bending:

- [ADR 0016](0016-web-initiated-enrichment.md) added `bills.last_enrichment_error`, a column written by the web process to record a failed enrichment attempt. It is not present in any JSONL envelope and cannot be reproduced by re-running the loader.
- [ADR 0017](0017-sqlite-schema-versioning.md) gave migrations a first-class home and, in its "What stays open" section, named the exact gap this ADR closes: *"A sibling amendment to 0002 noting this category ('derived store + small amount of web-runtime state that survives a JSONL rebuild') is worth writing but is out of scope for this ADR."*
- [ADR 0020](0020-bill-brief-llm-summary.md) introduces the first substantial member of the category: an LLM-generated **Bill Brief**, stored in `bill_briefs`. A brief is derived *from* the data, but it is non-deterministic (an LLM authored it), costs money to produce, and cannot be regenerated byte-for-byte from JSONL. Treating it as "rebuildable derived state" would mean a routine `concord index` re-run silently discards work the operator paid an API bill for.

The single word "derived" is now carrying two different contracts, and conflating them is a footgun: someone reasoning from "SQLite is disposable, just rebuild it" would wipe real, unrecoverable application state.

## Decision

Split the conceptual category of SQLite tables in two. The distinction is documentary and operational — there is **no** mechanical enforcement beyond the conventions below, and both kinds continue to live in the one SQLite file managed by `ensure_schema`.

**Mirror tables** — a deterministic projection of the JSONL canonical store. `proceedings`, `members`, `member_terms`, `bills` (its tier-1 + tier-2 columns), `bill_cosponsors`, `bill_actions`, `bill_subjects`, `bill_titles`, `bill_summaries`, `votes`, `vote_positions`, and the derived-index tables built on top of them (`chunks`, `chunks_fts`, `chunks_vec`, `chunking_status`, `bills_fts`, `members_fts`, `member_party_unity`). The 0002 contract holds unchanged: drop them, re-run the relevant Stage 1/2 pass, and you get a byte-identical result. They are in the blast radius of any schema/chunking/embedding change, and that is fine because they are free to rebuild.

**Record tables** — Concord's own application state and generated artifacts, which originate *here* rather than upstream and are **not** reconstructable from JSONL by re-running the pipeline:

- `bill_briefs` (ADR 0020) — LLM-generated executive summaries.
- `bills.last_enrichment_error` (ADR 0016) — a record *column* on an otherwise-mirror table; the rest of the row is a mirror, this one column is web-runtime state. (Documented here; not worth splitting into its own table.)

The contract for record tables:

1. **A JSONL re-derivation must not destroy them.** The Stage 1 loaders UPSERT mirror rows and never `DROP`/`DELETE` record tables; `ensure_schema` only ever `CREATE … IF NOT EXISTS`. So the existing rebuild path (`concord load` / `concord index`) already preserves record state today, and any future "wipe and rebuild the mirror" tooling must scope its deletes to mirror tables only.
2. **They carry their own provenance.** A record row records enough to know what it was derived from and whether that basis has since moved. `bill_briefs` stores a `facts_hash` (the fact-pack it was generated against) plus the `model` and `prompt_version`, so the web layer can flag a brief stale when the underlying mirror data changes — see ADR 0020.
3. **They are not in the canonical store, and that is a deliberate asymmetry.** Mirror data has JSONL as its recovery point; record data does not. Losing `proceedings.db` loses zero mirror information (re-derive) but *does* lose record information. For the current deployment shape (a personal demo, one SQLite file) this is an accepted, documented risk, not an oversight — see below.

This ADR does not move record data into JSONL, does not add a second database, and does not introduce a naming-prefix scheme. It draws the line and writes down the contract; the mechanism stays the one file + the 0017 migration runner.

## Consequences

**Trade-offs accepted:**

- **"SQLite is fully rebuildable from JSONL" is no longer unconditionally true.** It is true for mirror tables (the overwhelming majority of bytes) and false for record tables. Anyone reasoning about backup/rebuild must now know which kind they're touching. This ADR + the inline schema comments on `bill_briefs` are the documentation that makes the distinction discoverable rather than surprising.
- **Record data needs a real backup story eventually.** Today there is none beyond "don't delete the file." Acceptable for a single-user demo where a lost brief is one re-click and a few cents to regenerate. The moment record tables hold something a user would be upset to lose (saved annotations, curated collections), this ADR's "what stays open" becomes a must-do.
- **No enforcement.** A careless `DROP TABLE bill_briefs` followed by rebuild won't be caught by tooling. We rely on the convention + the FK/`IF NOT EXISTS` discipline rather than building a guard rail that the deployment shape doesn't yet justify.

**Things this buys:**

- **A stable vocabulary for future ADRs.** "Is this a mirror table or a record table?" is now a question with a defined answer and a defined contract, instead of an ambiguous appeal to the word "derived."
- **Permission to cache expensive generated artifacts in SQLite** without feeling like it violates 0002. The brief (ADR 0020) is the first beneficiary; future ones (cached cross-entity rollups, user-saved views) have a home and a rule set.
- **The rebuild path is unchanged and still safe.** Because the loaders already UPSERT and `ensure_schema` is already `IF NOT EXISTS`, no code change is required to honor contract (1) today — this ADR ratifies existing-and-correct behavior and forbids regressing it.

**What stays open:**

- **Durability for record tables.** A periodic dump of `bill_briefs` (and any future record table) to a sidecar file, or promoting genuinely precious record data into its own canonical store, is the migration when the stakes rise. Explicitly deferred.
- **A `--mirror-only` teardown flag.** If/when a "rebuild the derived store from scratch" command is added, it must delete mirror tables only. Noted here so that command is born correct.

## Rejected: a second database (or a record-store JSONL)

Putting record tables in their own SQLite file (or giving them a canonical JSONL of their own) would make the mirror/record split physical and the backup story crisper. Rejected for now: it doubles the connection/bootstrap plumbing and the `ensure_schema` surface for a single small table, and the 0017 migration runner already manages one file cleanly. When durability becomes a requirement (above), a dedicated record store is the natural shape — recording the option here so a future ADR can adopt it without re-deriving the reasoning.

## Rejected: keep calling everything "derived"

The zero-effort option: leave 0002's language alone and rely on tribal knowledge that "some columns are special." Rejected because the whole point of the ADR log is to make non-obvious decisions legible, and "this table looks derived but will be silently destroyed if you treat it as derived" is precisely the kind of trap that needs to be written down.
